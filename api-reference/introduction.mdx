# Sindre API Reference

The Sindre API provides comprehensive access to meeting intelligence, task management, and integration capabilities. This reference covers all available endpoints, request/response formats, and best practices.

## Base URL

All API requests should be made to:

```
https://api.sindre.com
```

For self-hosted installations, replace with your domain.

## Authentication

### JWT Token Authentication

All API requests require authentication using a JWT token from Supabase Auth in the `Authorization` header:

```javascript
const headers = {
  'Authorization': `Bearer ${SUPABASE_JWT_TOKEN}`,
  'Content-Type': 'application/json'
};
```

<Tabs>
  <Tab title="JavaScript">
    ```javascript
    const response = await fetch('https://api.sindre.com/api/meetings', {
      method: 'GET',
      headers: {
        'Authorization': `Bearer ${SUPABASE_JWT_TOKEN}`,
        'Content-Type': 'application/json'
      }
    });
    ```
  </Tab>
  <Tab title="cURL">
    ```bash
    curl -X GET \
      -H "Authorization: Bearer YOUR_JWT_TOKEN" \
      -H "Content-Type: application/json" \
      https://api.sindre.com/api/meetings
    ```
  </Tab>
  <Tab title="Python">
    ```python
    import requests
    
    headers = {
        'Authorization': f'Bearer {SUPABASE_JWT_TOKEN}',
        'Content-Type': 'application/json'
    }
    
    response = requests.get('https://api.sindre.com/api/meetings', headers=headers)
    ```
  </Tab>
</Tabs>

### Getting Your JWT Token

<Steps>
  <Step title="Authenticate with Supabase">
    Use Supabase Auth to sign in users and obtain JWT tokens
  </Step>
  <Step title="Extract Token">
    Get the access token from the Supabase session: `session.access_token`
  </Step>
  <Step title="Include in Requests">
    Pass the JWT token in the Authorization header for all API calls
  </Step>
</Steps>

<Warning>
  **Keep JWT tokens secure!** These tokens provide access to user data. Never expose them in client-side code or commit them to version control. Handle token refresh automatically using Supabase client libraries.
</Warning>

## API Conventions

### RESTful Design

The Sindre API follows REST conventions with predictable endpoint patterns:

<AccordionGroup>
  <Accordion icon="list" title="Resource Collections">
    ```
    GET    /api/meetings     # List all meetings
    POST   /api/meetings     # Create a new meeting
    ```
  </Accordion>
  <Accordion icon="file" title="Individual Resources">
    ```
    GET    /api/meetings/:id # Get specific meeting
    PUT    /api/meetings/:id # Update meeting
    DELETE /api/meetings/:id # Delete meeting
    ```
  </Accordion>
  <Accordion icon="filter" title="Filtered Collections">
    ```
    GET /api/tasks?status=pending        # Filter by status
    GET /api/tasks?meetingId=123         # Filter by meeting
    GET /api/meetings?limit=10&offset=20 # Pagination
    ```
  </Accordion>
  <Accordion icon="link" title="Nested Resources">
    ```
    GET    /api/tasks/:taskId/comments           # Get task comments
    POST   /api/tasks/:taskId/comments           # Create comment
    PUT    /api/tasks/:taskId/comments/:commentId # Update comment
    ```
  </Accordion>
</AccordionGroup>

### HTTP Methods

<CardGroup cols={4}>
  <Card title="GET" icon="download">
    **Retrieve data**
    - Safe and idempotent
    - No request body
    - Returns resource data
  </Card>
  <Card title="POST" icon="plus">
    **Create resources**
    - Request body required
    - Returns created resource
    - Status: 201 Created
  </Card>
  <Card title="PUT" icon="pen">
    **Update resources**
    - Full resource replacement
    - Request body required
    - Status: 200 OK
  </Card>
  <Card title="DELETE" icon="trash">
    **Remove resources**
    - No request body
    - Status: 204 No Content
    - Returns success message
  </Card>
</CardGroup>


## HTTP Status Codes

<AccordionGroup>
  <Accordion icon="check-circle" title="Success Codes (2xx)">
    - **200 OK** - Request successful, data returned
    - **201 Created** - Resource created successfully
    - **204 No Content** - Request successful, no data to return
  </Accordion>
  <Accordion icon="warning" title="Client Error Codes (4xx)">
    - **400 Bad Request** - Invalid request format or parameters
    - **401 Unauthorized** - Missing or invalid authentication
    - **403 Forbidden** - Access denied for this resource
    - **404 Not Found** - Resource does not exist
    - **429 Too Many Requests** - Rate limit exceeded
  </Accordion>
  <Accordion icon="x-circle" title="Server Error Codes (5xx)">
    - **500 Internal Server Error** - Unexpected server error
    - **502 Bad Gateway** - Upstream service error
    - **503 Service Unavailable** - Service temporarily unavailable
  </Accordion>
</AccordionGroup>

  <Accordion icon="clock" title="Rate Limits">

## Pagination

Large collections are paginated for performance:

### Query Parameters

```
GET /api/meetings?limit=20&offset=40
```

<ParamField query="limit" type="integer" default="50">
  Number of items to return (max 100)
</ParamField>

<ParamField query="offset" type="integer" default="0">
  Number of items to skip
</ParamField>

### Pagination Response

```json
{
  "data": [...],
  "pagination": {
    "total": 150,
    "limit": 20,
    "offset": 40,
    "has_more": true,
    "next_offset": 60,
    "prev_offset": 20
  }
}
```
